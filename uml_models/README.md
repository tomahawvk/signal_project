## UML Models

## 1. Alert Generation System
This subsystem separates alert evaluation, threshold management, and alert dispatching. AlertGenerator receives real-time patient data and asks ThresholdManager for the rules assigned to that patient. Each AlertRule represents one personalized condition, such as saturation below 90% or heart rate above a patient-specific limit. If a rule is triggered, the generator creates an Alert containing the patient ID, condition, severity, and timestamp. The alert is then passed to AlertManager, which routes it to staff through one or more StaffNotifier implementations. DataStorage is linked because alert evaluation may need recent patient history, not only the current measurement. Access is limited by responsibility: generators evaluate data, threshold classes manage rules, and only the manager handles notification routing.

## 2. Data Storage System
This subsystem models secure storage and retrieval of incoming health data. DataStorage is the main entry point and owns patient-specific record stores. Each PatientRecordStore contains many timestamped PatientData objects for one patient. The version field supports traceability when records are corrected or replaced. DataRetriever handles query requests from medical staff instead of exposing storage internals directly. Before returning records, DataStorage checks AccessController to ensure the requesting User has permission to read the selected patient’s data. RetentionPolicy defines when old records should be deleted. This keeps storage, access control, querying, and deletion rules separate, which improves maintainability and supports privacy requirements.

## 3. Patient Identification System
This subsystem links simulator patient IDs to real hospital patient records. PatientIdentifier is responsible for matching incoming PatientData to a HospitalPatient. It asks HospitalPatientRepository to retrieve the hospital record associated with the simulator ID. IdentityManager validates the match and handles cases where no patient exists, duplicate mappings appear, or the incoming ID looks suspicious. Such problems are represented as IdentityIssue objects with a reason and timestamp. Sensitive fields such as name and medical history stay inside HospitalPatient and should only be accessed by classes that need identity information. This design keeps matching logic separate from storage and alert generation, reducing the chance that private patient details are exposed unnecessarily.

## 4. Data Access Layer
This subsystem isolates external input from the rest of the CHMS. DataListener defines a shared interface for starting and stopping data input. TCPDataListener, WebSocketDataListener, and FileDataListener implement the same interface, so the rest of the system does not need to know how the data arrived. Each listener passes raw text to DataParser, which converts formats such as CSV, JSON, or log lines into a standard PatientData object. DataSourceAdapter receives parsed data and forwards it to DataStorage. This design supports change: a new data source can be added later by creating another DataListener implementation without changing storage, alerting, or identification logic.
